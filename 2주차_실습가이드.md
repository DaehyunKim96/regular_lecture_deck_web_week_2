# 2주차 실습 가이드: Claude Cowork로 회의록 자동화 + 예약 작업

## 0. 1주차 리마인드 (연결고리)

1주차: ChatGPT + Gmail 플러그인 → 메일 읽기 → 요약 → Slack/Notion 전송 → 이를 ChatGPT 스킬로 만들고 → 예약(scheduled) 작업으로 전환.

2주차는 같은 구조("입력 → 처리 → 출력 → 반복 자동화")를 **Claude Cowork**로 동일하게 구현해본다. 도구만 바뀔 뿐 사고방식은 동일하다는 점을 먼저 짚고 시작.

---

## 1. 예약 작업(스케줄)이 PC/노트북이 꺼져도 도는가?

**결론: 조건부.**

- Cowork의 예약 작업은 기본적으로 **원격(클라우드)에서 실행**된다. 즉 PC가 잠자기 모드거나 완전히 꺼져 있어도, 심지어 Claude 데스크톱 앱을 닫아둔 상태여도 정해진 주기에 그대로 실행된다.
- 단, **예외**가 있다: 예약 작업이 **로컬 파일이나 로컬 앱(내 컴퓨터의 특정 폴더 등)에 접근해야 하는 경우**엔 그 작업이 "로컬 전용"으로 분류되어, **PC가 켜져 있고 Claude 데스크톱 앱이 실행 중일 때만** 동작한다. (컴퓨터가 잠들어 있으면 건너뛰고, 다시 켜지거나 앱을 열면 그때 실행됨)

즉 오늘 이 세션처럼 "사용자가 선택한 로컬 폴더"를 입력/출력 위치로 쓰면 → 로컬 전용 모드가 되어 PC를 켜둬야 한다.

**완전 자동화(PC 꺼져도 동작)를 원한다면:**
입력과 출력을 로컬 폴더가 아니라 **클라우드 커넥터**(Slack, Notion, Google Drive 등)로 구성하면 된다. 1주차 ChatGPT 실습이 PC와 무관하게 돌아갔던 이유도 Gmail → Slack/Notion이 전부 클라우드 서비스였기 때문. 2주차도 같은 원리로, 회의 메모를 로컬 파일 대신 Notion 페이지나 Google Drive에 저장해두고, 결과물도 Notion/Slack으로 보내도록 구성하면 노트북을 꺼도 매주 알아서 돌아가는 자동화가 된다.

출처: [Schedule recurring tasks in Claude Cowork – Claude Help Center](https://support.claude.com/en/articles/13854387-schedule-recurring-tasks-in-claude-cowork)

## 2. 회사에서 회의록은 보통 어디에 올라가나 → 어떤 커넥터를 쓸까

회의록이 쌓이는 위치는 조직마다 다르고, Cowork에서 바로 붙는 사전 구축 커넥터(pre-built connector)도 정해져 있다. 실습 전에 "우리 회사는 어디에 회의록을 올리는지" 먼저 손을 들어보게 하고, 아래 표로 매칭해주면 된다.

| 회사에서 회의록을 올리는 곳 | 대응하는 Cowork 커넥터 | 비고 |
|---|---|---|
| Notion (위키/문서 중심 조직, 스타트업에 흔함) | **Notion 커넥터** | 검색(search)·조회(fetch)·페이지 생성(create-pages)·업데이트(update-page) 지원 |
| Google Docs / 공유 드라이브 (Google Workspace 조직) | **Google Drive 커넥터** | 파일 검색·읽기·생성 지원 |
| Outlook/Teams/OneNote/SharePoint (공공기관·전통적 대기업에 흔함) | **Microsoft 365 커넥터** | Teams 회의 메모, OneNote, SharePoint 문서 연동 |
| Confluence (개발 조직, Jira와 같이 쓰는 경우) | **Atlassian(Rovo) 커넥터** | Confluence 페이지 검색·조회 |
| 그 외 사내 그룹웨어/카카오워크·잔디 등 | 공식 커넥터 없음 → 회의 메모를 텍스트로 붙여넣거나 파일 업로드해서 진행 | 이 경우 예약 작업은 "로컬 전용"이 되어 PC가 켜져 있어야 함 |

결과 공유(업무보고/요약 전달)는 대부분의 조직에서 **Slack** 또는 **Teams**로 수렴하므로, Slack 커넥터는 저장소가 무엇이든 거의 항상 같이 쓰게 된다.

**이번 실습에서 쓸 조합: Notion(저장) + Slack(공유)**
1주차에 이미 Gmail→Slack/Notion 흐름을 다뤄봤기 때문에 학습 곡선이 낮고, 두 커넥터 모두 Cowork에 바로 연결 가능하다. Google Workspace나 MS 365를 쓰는 조직 수강생에게는 저장소만 Notion → Google Drive / Microsoft 365로 바꾸면 동일한 흐름이 그대로 적용된다는 점을 짚어준다.

## 3. 실습 순서

### Step 1. 커넥터 연결 확인 (5분)
- Cowork 사이드바에서 Notion, Slack 커넥터가 연결(Connect)되어 있는지 확인. 안 되어 있으면 그 자리에서 연결
- 자신의 회사가 Notion을 안 쓴다면, 표를 참고해 Google Drive/Microsoft 365/Confluence 중 해당하는 커넥터로 대체 — 이후 스텝은 동일한 논리로 진행

### Step 2. 사전 준비 — Notion 회의록 데이터베이스 만들기 (5분)
1. Notion에 페이지를 하나 만들고, 그 안에 데이터베이스(테이블)를 하나 생성
2. 속성에 최소 "회의일자"(날짜 속성)를 포함 — 그 외 참석자/프로젝트명 등은 자유롭게 추가
3. 회의가 끝날 때마다 이 데이터베이스에 새 행(row)을 추가하고, 각 행이 하나의 회의록 페이지가 되도록 구성 (실습용으로 2~3건을 미리 채워둔다)

→ 이렇게 만들어두면 이후 모든 실습(업무보고 실습, 블로그 실습)이 "이 데이터베이스에서 최신 회의록을 찾는다"는 동일한 출발점을 공유하게 된다.

### Step 3. 실습 A — 회의록 → 업무보고 변환 (단발 실행, 20분)
1. Cowork에 다음과 같은 형태로 요청: "Notion에서 [데이터베이스명] 데이터베이스를 조회해서(search/fetch) 회의일자가 가장 최근인 행을 찾고, 그 페이지를 열어 내용을 확인해줘"
2. 확인한 최신 회의록을 회의록 양식(일시/참석자/안건/결정사항/액션아이템)으로 정리해 새 Notion 페이지로 저장(create-pages)하도록 이어서 요청
3. 정리된 회의록을 업무보고 양식으로 재구성하도록 이어서 요청
4. 완성된 업무보고를 지정 Slack 채널로 전송(slack_send_message)하도록 요청

→ 이 단계는 커넥터 기반이라 PC를 꺼도 나중에 예약 작업으로 그대로 옮겨갈 수 있다는 점을 짚어준다.

**프롬프트 예시 (가장 최신 회의록만 읽어서 인사이트 뽑아 Slack으로 보내기)**

바로 위 1~4번을 한 번에 요청할 수도 있지만, 처음 시연할 때는 "최신 회의록만 읽고 인사이트만 뽑아 Slack에 보내기"처럼 범위를 좁혀 아래 프롬프트로 먼저 결과를 보여주면 이해가 쉽다.

```
Notion의 "실습 회의록 모음" 데이터베이스에서 회의 날짜가 가장 최근인 회의록을 찾아서
그 페이지 내용을 읽어줘.

그 내용을 바탕으로 아래 포맷을 그대로 따라서 정리하고, Slack의 [채널명] 채널로 보내줘.
포맷의 대괄호 [ ] 부분만 실제 내용으로 채우고, 이모지·볼드·불릿 기호는 그대로 유지해줘.

---
📋 *[회의 제목]* ([회의 날짜])

*✅ 핵심 결정사항*
• [이미 결론이 난 것만, 최대 3줄]

*📌 담당자별 액션아이템*
• *[담당자]*: [할 일] (마감 기한이 언급되어 있으면 함께 표시)

*⏳ 아직 결론 안 난 이슈*
• [다음 회의로 넘어간 논의 중인 사항. 없으면 이 항목은 생략]
---
```

이렇게 프롬프트에 포맷을 고정으로 박아두면, 회의록 내용이 바뀌어도 Slack 메시지 구조(소제목·이모지·볼드·불릿 위치)는 매번 똑같이 유지된다. 예약 작업으로 전환했을 때 특히 중요한 부분 — 매주 결과물의 모양이 들쭉날쭉하면 팀원들이 못 알아보기 때문.

**Slack 포맷 관련 참고사항**
- Slack은 일반 마크다운이 아니라 mrkdwn 문법을 쓴다: 굵게는 `*텍스트*` (별표 1개), 제목(`#`)은 지원 안 되므로 이모지+볼드로 소제목처럼 흉내낸다, 불릿은 `•` 문자를 직접 사용
- 위 포맷을 그대로 프롬프트에 넣어두면 Cowork가 Slack 전송 시 이 문법을 그대로 써서 보내준다

**7/20(가장 최근) 회의록으로 실제 채웠을 때 예시**

```
📋 *주간 정기회의 - 7월 3주차* (2026.07.20)

*✅ 핵심 결정사항*
• 실습 DB는 회의일자 속성이 있는 테이블 형태로 구성
• 저장은 Notion, 공유는 Slack으로 확정
• 예약 작업 프롬프트에 "외부 공개 가능 여부" 판단 기준 포함하기로 함

*📌 담당자별 액션아이템*
• *이*: 실습 DB에 샘플 데이터 채우기
• *최*: 슬라이드 표/다이어그램 반영
• *수강*: 최종 일정 조율

*⏳ 아직 결론 안 난 이슈*
• 커넥터 매핑표에 Drive/MS365/Confluence까지 포함할지 추가 논의 필요
```

이 프롬프트로 먼저 "읽기 → 인사이트 추출 → 전송"까지 감을 잡게 한 뒤, Step 3의 1~4번(회의록 정리 → 업무보고 재구성 → Notion 저장 → Slack 전송)으로 범위를 넓혀가면 난이도 곡선이 완만해진다.

### Step 4. 실습 A를 예약 작업으로 전환 (10분)
1. Step 3에서 쓴 프롬프트를 그대로 `/schedule` 또는 "Scheduled" 메뉴에서 예약 작업으로 등록
2. 프롬프트의 핵심 조건("회의일자가 가장 최근인 행을 찾아서")은 그대로 유지 — 데이터베이스에 새 행이 쌓일 때마다 자동으로 최신 것을 골라내는 구조이기 때문에 별도 트리거 없이도 매번 최신 회의록을 처리
3. 주기 설정(예: 매주 월요일 오전 9시) 후 저장 — Cowork 예약 작업은 시간 주기 기반이라 "새 행 추가 시 즉시 실행" 같은 이벤트 트리거는 지원하지 않는다는 점 안내
4. 출력은 Notion 페이지 생성 + Slack 메시지 전송으로 그대로 유지

### Step 5. 실습 B — 회의록 → 블로그 포스트 초안 자동 작성 (단발 실행, 20분)
Step 2에서 만든 것과 **같은 Notion 회의록 데이터베이스**를 그대로 사용하되, 산출물만 업무보고 대신 블로그 포스트로 바꾸는 실습.

1. Cowork에 요청: "Notion에서 [데이터베이스명] 데이터베이스를 조회해서 회의일자가 가장 최근인 행을 찾고, 그 페이지를 열어 내용을 확인해줘" (Step 3-1과 동일)
2. 회의 내용 중 외부에 공개해도 되는 부분(프로젝트 진행상황, 인사이트, 제품 업데이트 등)만 골라 블로그 포스트 초안으로 재작성하도록 요청 — 어조/문체(예: 회사 공식 블로그 톤, 친근한 뉴스레터 톤)를 구체적으로 지정하는 실습 포함
3. 완성된 초안을 Notion에 "블로그 초안" 페이지로 별도 저장(create-pages)
4. 콘텐츠 담당자에게 리뷰를 요청하는 형태로 Slack 채널에 알림 전송(slack_send_message) — "새 블로그 초안이 올라왔습니다" + 링크

### Step 6. 실습 B를 예약 작업으로 전환 (10분)
1. Step 4와 동일한 방식으로 `/schedule` 등록
2. 다만 모든 회의록을 블로그로 만들 필요는 없으므로, 프롬프트에 "외부에 공개 가능한 내용이 있는 회의록일 때만 초안을 작성하고, 그렇지 않으면 건너뛴다"는 판단 기준을 명시하는 것이 핵심 — 이런 조건부 판단 기준을 프롬프트로 정의하는 법이 이번 확장 실습의 포인트
3. 주기 설정(예: 매주 월요일 오전 9시, 업무보고와 같은 시간대에 같이 실행되도록) 후 저장

### Step 7. 실행 결과 확인 및 트러블슈팅 (5분)
- "Scheduled" 탭에서 실행 내역(성공/실패/스킵) 확인하는 법
- 로컬 파일을 입력으로 잘못 지정했을 때 "PC가 꺼져 있어 건너뜀" 상태가 뜨는 경우 확인 → 커넥터 기반으로 재설정

### Step 8 (여유 시간). 다른 자동화 아이디어에 적용
- 설문에서 나온 다른 사례(이메일 분류, 일정 리서치, 자료취합)에도 같은 "저장소를 클라우드 커넥터로 바꾸면 완전 자동화된다"는 원칙을 적용해보는 짧은 토론

## 4. 강의 포인트 요약

- 1주차와 2주차는 "도구(ChatGPT vs Claude)만 다를 뿐 구조는 동일"하다는 것을 계속 상기시키기
- "예약 작업 = 무조건 PC 꺼도 됨"이 아니라 **입력/출력이 클라우드냐 로컬이냐에 달려있다**는 점이 이번 실습의 핵심 개념
- 회의록 실습은 "추출(회의 메모→구조화) → 산출물(보고서/블로그) → 공유(Notion/Slack)"의 축소판 파이프라인임을 짚어주기
